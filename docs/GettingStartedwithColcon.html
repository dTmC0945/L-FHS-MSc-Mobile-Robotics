9.1.1 The Structure of a Package

A ROSRobot Operating System 2Robot Operating System 2 workspace is a directory with a defined structure. Commonly there is a src subdirectory. (...) This kind of directory usually exists for development of applications written in C or C++. Within this subdirectory is where the source code of ROSRobot Operating System 2Robot Operating System 2 packages will be located.

When this directory is created, it will be empty.

The primary job of colcon is to create the structure for, compile and build these source files. By default it will create the following directories with the same hierarchy of the src directory:

build

Where intermediate files are stored. For each package a subfolder will be created in which e.g., CMake (...) a free, cross-platform, software development tool for building applications via compiler-independent instructions and allows automate testing, packaging and installation. install

Where each package will be installed to. By default each package will be installed into a separate subdirectory.

log

Contains various logging information about each colcon invocation, for use in error-checking and debugging purposes.

For anyone who’ve worked with ROS 1 and catkin, there is no devel directory.

9.1.2 Sourcing an Underlay

It is important that we’ve sourced the environment for an existing ROSRobot Operating System 2Robot Operating System 2 installation which will provide our workspace with the necessary build dependencies for the example packages. This is achieved by sourcing the setup script provided by a binary installation or a source installation. (...) i.e. another, colcon workspace.

We call this environment an underlay.

Our workspace, ros2\_ws, will be an overlay on top of the existing ROSRobot Operating System 2Robot Operating System 2 installation.

In general, it is recommended to use an overlay when we plan to iterate on a small number of packages, rather than putting all of our packages into the same workspace. This allows easy debugging if things go bad.

9.1.3 Building the Workspace

In the root of the workspace, run colcon build. Since build types such as ament_cmake do NOT support the concept of the devel space and require the package to be installed, colcon supports the option --symlink-install. This allows the installed files to be changed by changing the files in the source space (...) e.g., Python files or other non-compiled resources for faster iteration. 

colcon build --symlink-install --executor sequential

Information: Limitations of Virtualisation

The way we are currently building is NOT the official way as we had to add --executor sequential option. We are adding this as by default, the colcon processes/compiles packages parallel to speed up the compilation time. If we were to have ROSRobot Operating System 2Robot Operating System 2 installed on native hardware rather than a docker container we may not have needed this additional option. For more additional arguments, have a look at here.

Information: Overlay v. Underlay

As a final clarification on these two (2) concepts, let’s look at them in a bit more detail:

  • An underlay is the core ROSRobot Operating System 2Robot Operating System 2 installation that provides the foundational packages and environment for your ROSRobot Operating System 2Robot Operating System 2 development. In our case it is Humble.

  • An overlay is the secondary workspace where we can add new packages without interfering with the existing ROS 2 workspace that we’re extending.

9.1.5 Creating a Workspace

A workspace is a directory containing ROSRobot Operating System 2Robot Operating System 2 packages. Before using ROSRobot Operating System 2Robot Operating System 2, it’s necessary to source our ROSRobot Operating System 2Robot Operating System 2 installation workspace in the terminal we plan to work in. This makes ROSRobot Operating System 2Robot Operating System 2’s packages available for us to use in that terminal.

We also have the option of sourcing an “overlay”, which is a secondary workspace where we can add new packages without interfering with the existing ROSRobot Operating System 2Robot Operating System 2 workspace that we’re extending, or “underlay”.

Our underlay must contain the dependencies of all the packages in our overlay.

Packages in our overlay will override packages in the underlay.

It’s also possible to have several layers of underlays and overlays, with each successive overlay using the packages of its parent underlays.

There are some useful arguments for colcon build which are as follows:

--packages-up-to

builds the package we want, plus all its dependencies, but not the whole workspace, which saves time.

--symlink-install

saves us from having to rebuild every time we tweak python scripts.

--event-handlers console_direct+

shows console output while building, can otherwise be found in the log directory.

--executor sequential

processes the packages one by one instead of using parallelism.

9.1.11 Sourcing our Overlay

Before sourcing the overlay, it is very important we open a new terminal, separate from the one where we built the workspace. Sourcing an overlay in the same terminal where we built, or likewise building where an overlay is sourced, may create complex issues.

In the new terminal, source our main ROSRobot Operating System 2Robot Operating System 2 environment as the “underlay”, so we can build the overlay “on top of” it: 

source /opt/ros/humble/setup.bash

Time to go into the root of our workspace: 

cd ~/ros2_ws

In the root, source our overlay: 

source install/local_setup.bash

Information: Peculiarities in Sourcing

Sourcing the local_setup of the overlay will only add the packages available in the overlay to our environment. setup sources the overlay as well as the underlay it was created in, allowing us to utilise both workspaces. So, sourcing our main ROSRobot Operating System 2Robot Operating System 2 installation’s setup and then the ros2\_ws overlay’s local_setup, like we just did, is the same as just sourcing ros2\_ws’s setup, because that includes the environment of its underlay.

Now we can run the turtlesim package from the overlay:

9.1.13 Creating a Package

A package is an organizational unit for our ROSRobot Operating System 2Robot Operating System 2 code. If we want to be able to install our code or share it with others, then we’ll need it organized in a package. With packages, we can release our ROSRobot Operating System 2Robot Operating System 2 work and allow others to build and use it easily.

Package creation in ROSRobot Operating System 2Robot Operating System 2 uses ament as its build system and colcon as its build tool. We can create a package using either CMake or Python, which are officially supported, though other build types do exist which are not covered by the lecture.

The Anatomy of a Package
ROSRobot Operating System 2Robot Operating System 2 Python has its minimum required contents which are as follows:
package.xml

file containing meta information about the package

resource/<package_name>

marker file for the package

setup.cfg

is required when a package has executables, so ros2 run can find them

setup.py

containing instructions for how to install the package

<package_name>

a directory with the same name as our package, used by ROSRobot Operating System 2Robot Operating System 2 tools to find our package, contains __init__.py

Packages in a Workspace

A single workspace can contain as many packages as we want, with each in their own folder. We can also have packages of different build types in one workspace.

However, we cannot have nested packages.

Best practice is to have a src folder within your workspace, and to create our packages in there. This keeps the top level of the workspace clean so to say.z
